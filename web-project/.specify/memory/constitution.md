# 项目宪法

**项目名称**: Web 项目 - 数独游戏与待办事项应用  
**版本**: 1.0.0  
**批准日期**: 2024-12-19  
**最后修订日期**: 2024-12-19

## 概述

本项目是一个使用纯 Web 技术（HTML、CSS、JavaScript）构建的网页应用，包含数独游戏和待办事项管理功能。项目面向 Web 开发新手，代码包含详细的中文注释，便于学习和理解。

## 核心原则

### 原则 1: 代码可读性与注释完整性

**规则**:
- 所有代码文件必须包含详细的中文注释
- 注释应解释代码的功能、参数、返回值和使用场景
- 复杂逻辑必须提供算法说明和实现思路
- HTML 标签、CSS 属性、JavaScript 函数都应配有说明性注释

**理由**: 项目面向 Web 开发新手，详细注释有助于理解 Web 开发的基础概念和最佳实践。

### 原则 2: 纯前端技术栈

**规则**:
- 仅使用 HTML、CSS、JavaScript（原生，不使用框架）
- 数据持久化使用浏览器 localStorage API
- 不依赖外部库或框架（除非明确需要）
- 保持代码简洁，避免过度工程化

**理由**: 新手友好的技术栈，便于理解 Web 开发的基础原理，无需学习复杂的构建工具或框架。

### 原则 3: 渐进式功能实现

**规则**:
- 功能实现遵循从简单到复杂的顺序
- 每个功能模块应独立且可测试
- 优先实现核心功能，再添加增强特性
- 保持代码结构清晰，便于扩展

**理由**: 帮助新手理解功能开发的迭代过程，避免一次性实现过多复杂功能。

### 原则 4: 用户体验优先

**规则**:
- 界面设计简洁直观，符合现代 Web 设计规范
- 提供清晰的用户反馈（如确认对话框、错误提示）
- 支持键盘操作和移动设备交互
- 确保基本的功能可用性和响应性

**理由**: 良好的用户体验是 Web 应用的基础，即使是学习项目也应注重用户交互质量。

### 原则 5: 代码组织与维护

**规则**:
- 文件结构清晰：HTML、CSS、JavaScript 分离
- 使用语义化的 HTML 标签和 CSS 类名
- JavaScript 代码使用 IIFE 或模块化方式组织，避免全局污染
- 保持代码风格一致（缩进、命名等）

**理由**: 良好的代码组织有助于项目维护和团队协作，是专业开发的基础。

## 治理

### 修订程序

1. 任何对核心原则的修改需要经过讨论和确认
2. 版本号遵循语义化版本控制（MAJOR.MINOR.PATCH）
3. 修订后更新"最后修订日期"和版本号
4. 重大变更应在修订日志中记录

### 版本控制策略

- **MAJOR**: 核心原则的重大变更或移除
- **MINOR**: 新增原则或现有原则的扩展
- **PATCH**: 措辞澄清、格式调整等非实质性变更

### 合规审查

- 新代码提交前应检查是否符合所有核心原则
- 特别关注注释完整性和代码可读性
- 定期审查代码质量，确保符合项目标准
